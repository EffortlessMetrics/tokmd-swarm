## 💡 Summary
Fixed a bug where `cargo xtask jules-index` was missing items from `.jules/friction/done/` when generating `FRICTION_ROLLUP.md`. The indexing task now merges items from both `open/` and `done/` friction directories to provide a complete view.

## 🎯 Why
Memory and repo scaffolding documentation indicated that `FRICTION_ROLLUP.md` is generated by parsing both `.jules/friction/open/` and `.jules/friction/done/`. However, the generated index was missing the 12 closed items currently present in `.jules/friction/done/` due to a bug in the code that generated the file. Consolidating run learnings into generated indexes is target #2 for the Archivist persona.

## 🔎 Evidence
- `.jules/index/generated/FRICTION_ROLLUP.md` generated a table with no values.
- `ls -la .jules/friction/done/` showed there are 12 items.
- `.jules/index/generated/FRICTION_ROLLUP.md` generated from `xtask/src/tasks/jules_index.rs` only parsed from `.jules/friction/open/`.

## 🧭 Options considered
### Option A (recommended)
- Fix `jules-index` to include `.jules/friction/done/`. Update `xtask/src/tasks/jules_index.rs` to process both `.jules/friction/open/` and `.jules/friction/done/` into the Friction Index.
- Why it fits this repo and shard: Directly aligns with the memory rule for the workspace-wide shard and specifically addresses the priority of summarizing run packets into generated rollups.
- Trade-offs: Low risk, high value for tracking resolved friction items without touching product code.

### Option B
- Normalize `legacy_envelope.json` vs `envelope.json` across all runs.
- When to choose it instead: If the priority is data cleanup rather than fixing a known feature gap.
- Trade-offs: Requires rewriting history to correct `.jules/runs/*/legacy_envelope.json` which is not desired according to instructions.

## ✅ Decision
Option A. The current code in `xtask` misses an explicit requirement documented in memory to rollup the `/done/` friction items, leaving `FRICTION_ROLLUP.md` incomplete.
